# 활성-스코프 포커스 모드 — 인라인 뷰 풀 편집 (깊이 무관 기능 패리티)

> 사용자 최종 골: 깊이0(루트)의 모든 기능을 깊이n 하위영역에서도 사용. 방식(확정): **포커스 모드** — 영역 클릭 시 그 스코프가 활성(편집), 나머지 깊이는 읽기전용 dim(투명도/음영). 현재 인라인 뷰에서 안 되는 것: **드롭존 전체·그룹화·정렬 도구**.

## 핵심 통찰 (왜 포커스 모드인가)
- 앱엔 이미 **스코프 스택 네비게이션**이 있다: `scopes[]` + `activeIndex` + `navigateTo(scopes)` → `currentParentId = scopes[activeIndex].parentId` (page.tsx ~536). 어느 스코프로 navigate하면 `nodes`=그 스코프라 **드롭존/그룹/정렬/이동/연결/삭제/추가가 전부 네이티브로 동작**.
- 인라인 펼침 뷰가 패리티가 없는 이유: 자식이 `nodes`가 아니라 별도 `childNodes`(읽기·일부 라우팅)라서. 코어 기능(드롭존/그룹/정렬)은 `nodes` 기준이라 자식엔 안 먹음.
- 따라서 **활성 스코프를 `nodes`로 만들면(=navigate) 모든 기능이 공짜로 동작**. 자식별 개별 포팅 불필요(근본 해결).

## 모델 (피벗)
- **활성 스코프 1개** = 편집 대상 `nodes`/`edges`(풀 기능). 나머지(상위 체인 + 형제/하위 영역) = **읽기전용 dim 컨텍스트**.
- 영역(또는 dim된 다른 스코프 노드) 클릭 → 활성 전환(navigateTo 그 스코프).
- 자식은 "포커스 전 읽기전용 dim" → **현재의 자식별 라우팅(이동/삭제/연결/추가)을 대체**(포커스 후 네이티브 편집). 라우팅 코드는 유지하되 포커스 모델이 상위.

## 단계 (각 단계 브라우저 검증 + 커밋)
1. **활성 스코프 상태 + dim 렌더**: `activeScopeId`(기본=root). displayNodes에서 비활성 스코프 노드/엣지/영역을 opacity↓ + `selectable/draggable/connectable=false`(읽기전용). 활성 스코프만 정상·인터랙티브. 검증: 자식 dim 표시, 루트 편집 정상.
2. **상위 컨텍스트 렌더**: 활성 스코프가 자식일 때, 상위(조상) 스코프 노드를 fullGraph에서 읽어 dim 읽기전용 디스플레이로 활성 스코프 주변에 배치(buildScope 오프셋 역방향 — 활성 스코프 기준 좌표, 조상은 컨텍스트). 검증: 자식 활성 시 상위가 dim으로 보임.
3. **클릭으로 활성 전환**: 영역 라벨/빈 영역(또는 dim 노드) 클릭 → `navigateTo(그 스코프)` → `nodes`=그 스코프(풀 편집). 기존 펼침 chevron과 관계 정리(포커스가 펼침을 포함/대체). 검증: 영역 클릭 → 그 안에서 **드롭존·그룹·정렬·이동·연결·삭제·추가 전부 동작**(네이티브, 별도 작업 0).
4. **자식별 라우팅 정리**: 포커스 모델이 동작하면 Step 2~5(자식 이동/삭제/연결/추가)의 개별 라우팅은 "빠른 인라인 편집" 보조로 남기거나 제거. 회귀 검증.

## 위험·주의
- 큰 렌더링/상태 재구성. 동작 중인 5종 편집(`bc3cad5`까지) + 캔버스 고정(`87feada`)을 깨지 않게 단계별 + 브라우저 검증.
- 좌표: 활성 스코프 기준 좌표계로 전환 시 buildScope(현재 루트 기준)를 재구성. childOffsets/scopeOffsets 로직 재활용.
- 검증 함정(이 세션 실측): dev.db는 테스트마다 오염 → 단계 검증 전 `git checkout dev.db`+백엔드 재시작(실행 중 checkout은 readonly 유발). Playwright 연결 드롭은 flaky. 자식 RF 이벤트는 깨끗한 db에서 정상.
- 관련: [[inline-child-nodes-reactflow-gotcha]], `docs/superpowers/plans/2026-06-19-editable-inline-subprocess.md`(자식별 라우팅 완료분 + 드롭존 진단).
